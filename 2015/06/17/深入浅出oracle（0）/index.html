深入浅出oracle（0）

By shurrik

发表于 2015-06-17

　　最近接盘了一个项目，oracle，然后马上来了次12W条的update，还好比较顺利地完成了。生产数据库敢这样搞我也是醉了

　　还是比较迷糊，以前只用过mysql，于是在此总结一下。

　　首先说说oracle和mysql提交的区别，innodb存储引擎下，mysql进行数据操作(DML)时默认是自动commit的。而oracle一般需要commit（还有隐式提交）。

　　commit的提交针对的是:
　　DML，即对数据管理操作，比如insert、delete、update，select当然不用commit也行。
　　DDL，对数据结构定义，是不需要提交的，比如 create、alter、drop。

　　这个命令是将数据写到数据库中。如果不执行commit这个命令，那么在你这个session之外的其他session查询的数据是你修改数据之前的数据。而commit之后查询的是你修改后的数据。可以开两个sqlplus测试一下，一目了然。

　　其实commit和git的git commit类似，就是提交数据（这里是释放锁，不是锁表，insert以后commit之前是锁表的状态，如果不提交的话，那么这个表就被锁了，其他事务无法对该表进行操作，），锁有很多种，一般我们关注的都是DML操作产生的，比如insert，delete，update…都会同时触发表级锁和行级锁 。在未提交前你前面的操作更新的都是内存，没有更新到物理文件中。执行commit从用户角度讲就是更新到物理文件了，事实上commit时还没有写date file，而是记录了redo log file，要从内存写到data物理文件，需要触发检查点，由DBWR这个后台进程来写，这里内容有点多的，如果不深究的话就理解成commit即为从内存更新到物理文件。

　　commit通常是一个非常快的操作，花时间的反而是execute。它执行的时候，你不会有什么感觉。这点在数据库编程的时候很常用，当你执行DML操作时，数据库并不会立刻修改表中数据，这时你需要commit，数据库中的数据就立刻修改了，如果在没有commit之前，就算你把整个表中数据都删了，如果rollback的话，数据依然能够还原。听我这么说，你或许感觉commit没什么用，其实不然。当你同时执行两条或两条以上的sql语句时，问题就出现了。举一个例子，你去银行转账，你转的时候银行的数据库会update你银行账户里面的数据，同时对另一个人的账户也进行update操作。这两个程序都必须全部正确执行，才能commit，否则rollback。

　　你可能认为，一个事务越大(换句话说，它影响的数据越多)，commit需要的时间就越长。然而不是这样，不论事务有多大，commit的响应时间一般都很“平”(flat，可以理解为无高低变化)。这是因为commit并没有太多的工作去做，不过它所做的确实至关重要。

　　这一点很重要，之所以要了解并掌握这个事实，原因之一是：这样你就能心无芥蒂地让事务有足够的大小。原因之二是：12W条update我就分了4次完成…一种错误的观点认为分批提交可以节省稀有的系统资源，而实际上这只是增加了资源的使用。如果只在必要时才提交(即逻辑工作单元结束时)，不仅能提高性能，还能减少对共享资源的竞争。

　　分批提交commit的开销存在两个因素：
　　1.显然会增加与数据库的往返通信。如果每个记录都提交，生成的往返通信量就会大得多。
　　2.每次提交时，必须等待redo写至磁盘。这会导致“等待”。在这种情况下，等待称为“日志文件同步”(log file sync)。

　　所以在数据库中执行commit之前，困难的工作都已经做了。我们已经修改了数据库中的数据，所以99.9%的工作都已经完成。而commit之后是不能回滚rollback的，要恢复就得用闪回表flashback table，关于闪回表我了解不多，且听下回分解。
